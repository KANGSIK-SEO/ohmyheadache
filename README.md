# Mini NPU Simulator

## 실행 방법
1. Python 3.8 이상이 설치되어 있어야 합니다.
2. 프로젝트 루트에서 아래 명령으로 실행합니다.

```bash
python main.py
```

3. 모드를 선택합니다.
- `1`: 사용자 입력(3x3)
- `2`: data.json 분석

## 환경 변수(.env) 사용
민감 정보는 `.env` 파일에 저장하고, `.gitignore`에 의해 추적되지 않도록 설정했습니다.

예시:

```env
GITHUB_TOKEN=your_token_here
```

실제 토큰은 절대 Git에 커밋하지 마세요.

프로그램 시작 시 `.env` 파일이 있으면 자동으로 읽습니다.

## 초등학생도 따라하는 순서(Sequential)
1. 먼저 비교 기준 그림 2개를 준비해요.
- A 필터: 십자가
- B 필터: X
2. 입력 그림(패턴)도 같은 크기로 준비해요.
3. 같은 칸끼리 곱해요.
4. 곱한 값을 전부 더해요.
5. A 점수와 B 점수를 비교해요.
6. A가 더 크면 `Cross`, B가 더 크면 `X`예요.
7. 두 점수 차이가 아주 작으면 `UNDECIDED`로 처리해요.
8. data.json 모드에서는 케이스를 하나씩 반복해 같은 방법으로 판정해요.
9. 정답(expected)과 판정을 비교해 PASS/FAIL을 기록해요.
10. 마지막에 전체 통계와 실패 원인을 요약해요.

## 구현 요약
- MAC 연산: 2중 반복문으로 위치별 곱셈 후 누적 합산
- 라벨 정규화: `+`, `cross` -> `Cross`, `x` -> `X`
- 동점 처리: `abs(score_cross - score_x) < 1e-9`이면 `UNDECIDED`
- JSON 모드에서 케이스 키 `size_{N}_{idx}` 파싱 후 해당 크기 필터 자동 선택
- 스키마/크기 불일치가 발생해도 프로그램이 중단되지 않고 케이스 단위 FAIL 처리
- 성능 분석: 3x3, 5x5, 13x13, 25x25 각각 10회 반복 평균(ms) 출력

## 결과 리포트
본 구현은 실패를 단순히 오답으로 취급하지 않고, 원인을 데이터/스키마/수치정책으로 분리해 분석하도록 구성했다.
첫째, JSON 분석 단계에서 패턴 키가 `size_{N}_{idx}` 규칙을 벗어나면 로직 오류가 아니라 스키마 오류로 분류한다.
둘째, 입력 매트릭스의 행/열 크기 불일치나 숫자 타입 불일치는 데이터 품질 문제로 보고 케이스 단위 FAIL로 기록한다.
셋째, expected 라벨은 `+`/`x`와 같은 표현 차이로 실패하지 않도록 표준 라벨(`Cross`, `X`)로 정규화했다.
넷째, 필터 키도 `cross`/`x`를 표준 라벨로 변환하여 비교 기준을 일관화했다.
다섯째, 부동소수점 연산의 특성으로 미세한 차이가 생길 수 있으므로 epsilon(1e-9) 기준을 적용했다.
여섯째, epsilon 기반 비교가 없으면 사실상 동점인 케이스가 오분류되어 불필요한 FAIL이 증가할 수 있다.
일곱째, 본 정책을 통해 데이터 표현 차이와 수치 오차로 인한 허위 실패(false fail)를 줄여 재현성을 높였다.
여덟째, 시간 복잡도는 MAC 연산이 N x N 원소를 1회씩 방문하므로 O(N^2)다.
아홉째, 성능 표의 연산 횟수 열(N^2)은 이론 복잡도와 실측 시간 증가 경향을 연결하는 근거로 사용된다.
열째, 실제 측정에서도 3x3 대비 25x25가 더 큰 평균 시간을 보이며, 이는 연산 원소 수 증가와 직접적으로 대응한다.
열한째, 실패가 0개라면 이는 정규화/epsilon/스키마 검증 정책이 데이터 변동성을 흡수했기 때문이라고 해석할 수 있다.

## 재현성 체크
- 모드 1에서 예시 Cross/X 필터와 패턴을 입력하면 A/B 점수, 판정, 평균 연산 시간이 출력됩니다.
- 모드 2에서 data.json 분석 시 케이스별 PASS/FAIL, 총합(총/통과/실패), 실패 케이스 사유가 출력됩니다.
- 의도적 오류 케이스(`size_13_bad_schema`)는 프로그램 중단 없이 FAIL로 집계됩니다.
